Shorten time to market

now that software is eating the world.

In the digital era a key to success is to deliver better software faster. But it is well known that manually rolling out software is time consuming and errorprone. So in order to be able to deliver faster without jeopardizing quality it is needed to automate the complete deployment process. This is what continuous delivery is all about.

But the bigger a software system gets, the more complicated it is to automate its delivery process. To counterbalance it companies are replacing their monolithic systems with a set of loosely coupled microservices which can be deployed independently of each other.

Conway's law tells us that the architecture of a software system will quasi automatically mimic the organisation's communication structure. It explains why companies which are succeeding with microservices are organised in small autonomous agile teams.

Like the dogs pulling the sledge, each practice on its own is insuffecient. The true power comes from the combination.

Microservices

From a developer's point of view splitting up a complex system in smaller pieces makes a lot of sense. Smaller services are easier to understand, quicker to launch, hence boosting developer productivity. It also enables to choose the best tool/library for the job at hand inside each service. Again improving developer productivity, but also resulting in a higher pace of innovation.

A system built conform a microservices architecture is a distributed system. So engineering competences should be on par to handle the logging, tracing, monitoring, service discovery aspects of a distributed system.

Going the microservices road is not an easy undertaking, but it will offer a lot of rewards in return.

Continuous delivery

Using build pipelines to automatically deliver software systems

Often the major goal to adopt microservices, is to deliver software more frequently. Each service will have it's own build pipeline. It is important to ensure that each of the services can be rolled out independently. Otherwise the goal of faster delivery will not be fullfilled.

Rolling out software systems in containers is a technique which significantly decreases the complexity of setting up build pipelines. 'Microservices' and 'Containers' are in this way a match made in heaven.
